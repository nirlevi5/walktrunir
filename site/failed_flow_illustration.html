Failed Semantic Update - Table Flow

Only the suggested tables and their failure states

This mirrors the success flow and shows what happens in the tables when a semantic update fails at each phase. It intentionally excludes AWS services, queues, jobs, and external operational systems.

Failure table state

Early Failure

  • If the initial transaction fails, no new semantic_field_definitions row is committed.
  • No dataset_field_bindings rows are marked stale.
  • No propagation_events or propagation_tasks rows should exist for that failed attempt.

Discovery Failure

  • The definition update may already be committed, so stale bindings remain visible.
  • propagation_events.status becomes failed with an impact-assessment error.
  • No incomplete task set is allowed to drive recomputation.

Planning Failure

  • Task creation is transactional: either all propagation_tasks rows are inserted or none are.
  • The event is marked failed if a complete plan cannot be persisted.
  • Bindings stay stale until a retry or rollback event handles them.

Execution Failure

  • The failed dataset keeps its old datasets.current_materialization_uri.
  • Its task records failed, attempt_count, and error_message.
  • Downstream tasks stay blocked; the event cannot become completed.

Failure Coverage By Success Phase

  1. Publish phase: a failed definition/binding transaction rolls back, leaving all tables unchanged.
  2. Impact discovery phase: the semantic update exists, direct bindings are stale, and the propagation event is marked failed because the affected graph was not safely computed.
  3. Planning phase: the event exists, but task insertion fails atomically, so the event is failed and no partial task graph is used.
  4. Completion phase: a task fails before promotion, so the dataset read pointer and direct bindings remain on the previous committed state.
  5. Final status phase: any failed or blocked task prevents propagation_events.status from becoming completed.
