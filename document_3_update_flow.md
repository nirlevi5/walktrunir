# Document 3 — Update Flow

## Context

This document walks through the full lifecycle of correcting the `event_timestamp` semantic definition — from the moment an engineer submits the correction to the moment all affected datasets are recomputed and tenants are reading consistent data. Every step references the data model tables from Document 1 and the AWS services from Document 2.

The correction: `event_timestamp` was documented as `free_text_string`. It is actually `iso8601_date` (YYYY-MM-DD, UTC, no time component). The shared base data and physical schema do not change. Materialized derived datasets may still be rewritten, because their existing outputs were computed under the wrong semantic meaning.

---

## Phase 1 — Submitting the Correction

**The engineer calls:**
```
POST /definitions/event_timestamp/correct
{
  "semantic_type": "iso8601_date",
  "description": "ISO 8601 calendar date in UTC, no time component (YYYY-MM-DD). One value per event.",
  "time_semantics": { "timezone": "UTC", "precision": "day", "format": "YYYY-MM-DD" },
  "nullability_rule": "event_did_not_occur",
  "uniqueness_guarantee": false
}
```

**API Gateway** validates the request schema before the Lambda is invoked — field name exists, required fields are present, `semantic_type` is a known enum value.

**Metadata Writer Lambda** opens a single Aurora PostgreSQL transaction and performs three writes atomically:

1. **Insert a new row** into `semantic_field_definitions`:
   - New `definition_id` (uuid v4, generated by the Lambda)
   - `field_name = 'event_timestamp'`
   - `version = 2` (monotonically incremented per `(field_name, source_dataset_id)`)
   - `semantic_type = 'iso8601_date'`
   - `superseded_by = NULL` (this is now the current version)
   - `created_by = 'engineer@company.com'`

2. **Update the old row** in `semantic_field_definitions`:
   - Set `superseded_by = <new_definition_id>` on the version-1 row.
   - This is a single-row write. The old row is now retired — it is never deleted.

3. **Bulk-mark bindings as stale** in `dataset_field_bindings`:
   ```sql
   UPDATE dataset_field_bindings
   SET is_stale = true
   WHERE definition_id = <old_definition_id>
   ```
   This single statement marks every dataset that was computed against the wrong definition as stale. It may touch thousands of rows but is one atomic write within the transaction.

The transaction commits. At this point the database is consistent: the new definition exists, the old one is retired, and every dataset that used the old definition is marked stale. Nothing downstream has run yet.

**After the commit**, the Metadata Writer publishes a `DefinitionCorrected` event to the EventBridge custom bus:
```json
{
  "source": "walktru.semantic-catalog",
  "detail-type": "DefinitionCorrected",
  "detail": {
    "field_name": "event_timestamp",
    "old_definition_id": "<uuid-v1>",
    "new_definition_id": "<uuid-v2>",
    "triggered_by": "engineer@company.com",
    "initiated_at": "2026-06-23T10:00:00Z"
  }
}
```

The event is published after the transaction commits, not inside it. If the publish fails, the Lambda retries — safe because writing a definition that already exists is idempotent.

## Phase 2 — Impact Assessment (Finding the Affected Subgraph)

**EventBridge** routes the `DefinitionCorrected` event to the **DAG Traversal Lambda**.

The Lambda's job is to answer: "Which datasets are affected, and in what order must they be recomputed?" It answers this in three steps.

**Step 1 — Find directly affected datasets (depth 0):**
```sql
SELECT dataset_id FROM dataset_field_bindings
WHERE definition_id = <old_definition_id>
```
These are the datasets that directly reference `event_timestamp` with the wrong definition. Call this set `S₀`. These are the starting nodes for the graph traversal.

**Step 2 — BFS over `dataset_lineage` to find transitive dependents:**

Starting from `S₀`, the Lambda walks the `dataset_lineage` table downstream:
```sql
WITH RECURSIVE downstream AS (
  SELECT downstream_dataset_id, 1 AS depth
  FROM dataset_lineage
  WHERE upstream_dataset_id IN (<S₀ ids>)

  UNION ALL

  SELECT l.downstream_dataset_id, d.depth + 1
  FROM dataset_lineage l
  JOIN downstream d ON l.upstream_dataset_id = d.downstream_dataset_id
)
SELECT * FROM downstream
```
This returns every dataset that transitively depends on any depth-0 dataset, along with its graph depth. A billing deduplication dataset that depends on a DAU metrics dataset that depends on `event_timestamp` would appear at depth 2.

For very large graphs (>10k nodes), the Lambda pages the BFS in application code using an ElastiCache (Redis) frontier set, but the result is the same: a flat list of `(dataset_id, depth)` pairs covering the entire affected subgraph.

**Step 3 — Compute topological order and blocking relationships:**

For each affected dataset, the Lambda computes:
- `topological_depth`: its depth in the BFS above
- `blocked_by_task_ids`: the task IDs of its direct upstream dependencies *within the affected set*
- `recompute_reason`: `direct_semantic_binding` if the dataset directly bound to the old definition, otherwise `transitive_downstream`
- `binding_definition_ids_to_update`: `[old_definition_id]` for directly bound datasets, empty for downstream-only recomputations

A depth-2 dataset is blocked by its depth-1 predecessors, not by all depth-1 datasets — only those that are themselves in the affected subgraph. Datasets outside the affected subgraph are already correct and don't need to complete before their dependents can run.

**The Lambda then writes to the database:**

1. Inserts one row into `propagation_events`:
   - `old_definition_id`, `new_definition_id` from the event
   - `affected_dataset_count` = total count of tasks about to be created
   - `status = 'in_progress'`

2. Bulk-inserts one row into `propagation_tasks` per affected dataset:
   - `topological_depth` and `blocked_by_task_ids` as computed above
   - `recompute_reason` and `binding_definition_ids_to_update`
   - `pipeline_type` copied from the `datasets` table
   - `status = 'pending'` for depth-0 tasks, `'blocked'` for all others

3. Enqueues **only the depth-0 tasks** to SQS. Deeper tasks remain in the database as `blocked` — they are not touched until their upstream tasks complete.

**Why compute and insert all tasks up front, not dynamically?**
The full scope is visible before any work starts. An operator can query `propagation_tasks` right now to see exactly which 347 datasets (for example) will be recomputed and in what order. Dynamic task creation (create depth-1 tasks only after depth-0 finishes) would make the scope invisible until the propagation is halfway done.

## Phase 3 — Recomputation

Tasks are routed to one of two SQS queues based on `pipeline_type`:
- **Batch Task Queue** → Batch Task Starter Lambda
- **Streaming Task Queue** → Streaming Task Handler Lambda

### Batch Datasets

The **Batch Task Starter Lambda** picks up a message from the batch queue:

1. Sets `propagation_tasks.status = 'in_progress'`, records `started_at`.
2. Starts a Glue job (or EMR Serverless job run) parameterized with `dataset_id` and `new_definition_id`. The job reads the same upstream data sources but interprets `event_timestamp` according to the new definition — treating it as a proper date rather than a free-text string, which affects any date-range filtering, timezone handling, or aggregation logic in the transformation code.
3. The job writes a new materialization to a staging/versioned S3 prefix such as `s3://data-platform/{tenant_id}/datasets/{dataset_id}/propagations/{event_id}/`. The existing committed output remains readable until the Completion Handler promotes the new version.
4. The starter Lambda records the Glue/EMR job run ID and acknowledges the SQS message. The SQS message is only for starting the job; it is not used to hold the whole job open.
5. Glue publishes a job state-change event to EventBridge on completion (success or failure). That completion event triggers the next step.

**On success:** the Completion Handler is invoked.
**On failure:** `attempt_count` is incremented. Below the retry threshold, the failure handler enqueues a fresh start message. At the threshold (3 attempts), the task is marked `failed` and a CloudWatch alarm fires.

### Streaming Datasets

Streaming datasets require more surgical handling because a running consumer cannot simply be re-run from scratch — it has already been processing records in real time, and the checkpoint tracks its position in the stream.

The **Streaming Task Handler Lambda** executes a five-step protocol:

**1. Pause:**
Disable the Lambda event source mapping (for Kinesis-based consumers) or suspend the MSK consumer group. The consumer stops pulling new records. New events continue arriving in the stream; they accumulate but are not processed.

**2. Drain:**
Wait until the consumer's in-flight batch is fully committed. For Kinesis, this means monitoring the `IteratorAgeMilliseconds` CloudWatch metric until it drops to zero (the consumer has caught up to where it was paused). For MSK, consumer lag drops to zero.
The interim state (`consumer_id`, `pause_timestamp`, `drain_confirmed_at`) is written to a DynamoDB control-plane table. This is separate from Aurora because it's lightweight, ephemeral state — not semantic metadata.

**3. Update:**
Push the new semantic definition configuration to the consumer. In practice this means updating a Parameter Store value that the consumer reads at startup (e.g., `event_timestamp.semantic_type = iso8601_date`), or updating the Glue Schema Registry schema version that the consumer validates against.

**4. Historical backfill:**
Launch a bounded backfill from the immutable raw-event archive in S3 for the time range covered by the old semantic definition. The system should not rely on rewinding the live Kinesis/MSK stream for a year of history: stream retention may not be long enough, and replaying through the production consumer can duplicate side effects. The backfill writes corrected output to the same staging/versioned materialization prefix pattern used by batch jobs.

**5. Resume:**
Re-enable the event source mapping from the drained checkpoint, now interpreting `event_timestamp` as an ISO 8601 date. Events that arrived during the pause are processed once the consumer resumes. The task completes only after both the historical backfill has produced the corrected materialization and the live consumer is running on the new config.

**Why backfill from the definition's activation window and not just the pause time?**
The pause time only covers the records processed since the correction was initiated. But records processed before the correction were also interpreted under the wrong definition — the DAU metric aggregations, for example, may have bucketed dates incorrectly for months. Backfilling from the original definition's activation window ensures the entire historical period under the wrong definition is reprocessed.

## Phase 4 — Completion and Wave Progression

When a task completes (success signal from Glue EventBridge event or from the Streaming Task Handler), the **Completion Handler Lambda** runs:

1. **Within a single Aurora transaction:**
   - Sets `propagation_tasks.status = 'completed'`, `completed_at = now()` for the finished task.
   - Promotes the staging/versioned materialization for this dataset to the committed read pointer after validation.
   - For directly bound datasets only: updates matching `dataset_field_bindings` rows from the old definition to the new definition, sets `is_stale = false`, and records `bound_at = now()`. This is the moment the dataset officially transitions from the old semantic contract to the new one.
   - Increments `propagation_events.completed_dataset_count`.
   - If `completed_dataset_count = affected_dataset_count`: sets `propagation_events.status = 'completed'`, records `completed_at`.

2. **Release newly unblocked tasks:**
   The Completion Handler queries `propagation_tasks` for any task in `blocked` status whose `blocked_by_task_ids` are now all `completed`. For each newly unblocked task:
   - Sets its status to `pending`.
   - Enqueues it to the appropriate SQS queue (batch or streaming).

This "wave progression" is the core of topological execution. No task is ever enqueued until all its upstream dependencies within the affected subgraph have completed. The progression is driven entirely by the Completion Handler's database check — there is no central scheduler or Step Functions state machine.

## Partial Inconsistency — What Tenants See During Propagation

While propagation is in progress, some datasets have completed recomputation (`is_stale = false`, bound to the new definition) and some have not (`is_stale = true`). The system prevents half-written outputs from leaking by writing all recomputation results to staging/versioned locations and promoting a dataset's read pointer only after its task succeeds.

**The system manages it in three ways:**

**1. Topological guarantee:**
The recomputation order ensures that a downstream dataset is never recomputed before its upstream dependencies. At any point during propagation, a dataset bound to the new definition is guaranteed to have been computed from upstream datasets that are also already bound to the new definition (or are not in the affected subgraph). A mixed estate can exist between "waves," but it never means a depth-2 dataset is promoted to the new definition while its depth-1 upstream is still on the old one.

**2. Staleness visibility:**
The `is_stale` flag on `dataset_field_bindings` is visible to the data platform's query layer. The platform can expose this in three modes, configurable per tenant:
- **Warning mode (default):** Queries succeed against the last committed materialization but return metadata indicating the dataset was computed against an older semantic definition. Analysts see the result but are warned it may not reflect the corrected `event_timestamp` semantics.
- **Blocking mode (opt-in for high-stakes tenants):** Queries against stale datasets are rejected until recomputation completes. Appropriate for billing or compliance datasets where reading a stale version is worse than reading nothing.
- **Consistent-read mode:** Multi-dataset queries must use one compatible semantic generation for all affected inputs. If the query would mix old and new `event_timestamp` semantics, the planner either pins all inputs to the last compatible committed generation or rejects the query until propagation completes.

**3. No phantom consistency illusion:**
The system does not attempt to pretend propagation is complete until it actually is. `propagation_events.status` stays `in_progress` until every task has completed. The Status API (`GET /propagations/{event_id}`) reflects this in real time — operators and tenants can observe the rolling count of `completed_dataset_count / affected_dataset_count`.

## Rollback

Rollback is triggered manually by an operator — either because a task repeatedly failed, because the correction itself was wrong, or because tenants are reporting unexpected results.

**Detecting the need to roll back:**
- CloudWatch alarm on failed propagation tasks or DLQ message count > 0
- CloudWatch alarm on `PropagationLagSeconds` exceeding SLA threshold
- Manual call to `GET /propagations/{event_id}` showing stuck tasks

**Executing rollback:**
Rollback is not a special-case operation — it is a new correction in the reverse direction.

1. The engineer calls `POST /definitions/event_timestamp/correct` again, but this time with the old definition's semantics. The Metadata Writer inserts a version-3 row (which is semantically identical to version 1), sets `superseded_by` on the version-2 row, marks all bindings to version 2 as stale, and creates a new `propagation_events` row with `rollback_event_id` pointing to the original event.

2. The same EventBridge → DAG Traversal → SQS → Task Handlers → Completion Handler pipeline runs, recomputing all affected datasets against version 3 (the restored original semantics).

**Why this works:**
The shared base data was never modified. The Glue and streaming jobs wrote new materialized outputs, but those outputs are stored in tenant-scoped, versioned S3 prefixes. The original pre-correction outputs are not deleted until the new outputs are verified. Because `semantic_field_definitions` rows are immutable in their semantic contents, version 1 still exists in full detail — rollback means creating a new semantic version with the prior meaning and promoting materializations computed against it.

**Partial rollback:**
Rolling back a single tenant without affecting others is supported. The Metadata Writer scopes its `UPDATE dataset_field_bindings SET is_stale=true` to a specific `tenant_id`, and the DAG Traversal Lambda creates tasks only for that tenant's affected datasets. Other tenants' propagation continues uninterrupted.

## Tradeoffs

**Correctness vs cost — eager recomputation**

Every affected dataset is recomputed immediately, including datasets that tenants may not query for weeks. This is expensive: potentially hundreds of Glue jobs and streaming replays triggered by a single correction. The alternative — lazy recomputation, where a stale dataset is only recomputed when a tenant actually queries it — would save compute cost. But lazy recomputation turns the propagation into an unbounded, invisible process with no clear completion state. It makes rollback harder (you don't know what's been updated and what hasn't), and it means billing and compliance datasets can silently serve stale data. For a platform where `event_timestamp` feeds billing deduplication and identity joins, the risk of silent staleness outweighs the compute cost.

**Versioned definitions vs in-place updates**

Definitions are immutable in their semantic contents: each correction adds a new row, and old meaning/units/null rules are never overwritten. The alternative — updating the definition row in place — is simpler to implement but eliminates the ability to ask "what definition was this dataset computed against?" In-place updates also make rollback destructive: you've overwritten the thing you'd need to restore. Immutable versioning means rollback is a metadata-driven propagation, not a data recovery operation. The cost is slightly more storage (one row per definition version) and a marginally more complex query for "what is the current definition" — both trivially acceptable.

**Eager task set creation vs dynamic task spawning**

All `propagation_tasks` rows are computed and inserted at event creation time (before any task runs). The alternative is dynamic spawning: a task creates its downstream child tasks on completion. Dynamic spawning is simpler to implement but opaque: operators cannot see the full scope of work until the propagation has partially completed, and there is no way to estimate how long it will take. Up-front creation gives full observability immediately, makes it easy to catch DAG cycles or unexpectedly large impact sets before any recomputation starts, and simplifies rollback (there is always a complete task list to reason about).
