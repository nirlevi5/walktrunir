Document 2 — AWS Architecture

Services for semantic propagation · storage · orchestration · tenant isolation

Architecture Overview

Component Breakdown

DB
Aurora PostgreSQL (Serverless v2)

Single source of truth for all five metadata tables. One writer instance + one read replica; DAG traversal and status-check reads go to the replica. Deployed in a private VPC subnet; Lambdas connect via VPC config. Credentials in Secrets Manager with automatic rotation.

Why PostgreSQL, not DynamoDB: the bulk staleness sweep must be atomic with the definition insert — PostgreSQL gives this in one transaction; DynamoDB transactions cap at 100 items. time_semantics is JSONB, natively queryable. Lineage traversal is a recursive CTE; DynamoDB would require BFS in application code with N round-trips. Serverless v2 scales ACUs to zero when propagation is idle.
λ
API Gateway + Lambda: Metadata Writer

Entry point for definition corrections. API Gateway validates the request schema before the Lambda is invoked. The Lambda wraps all three writes — insert new definition, set superseded_by, bulk-update is_stale — in a single Aurora transaction. EventBridge publish happens after commit and is safe to retry (idempotent against an existing definition_id).

Why one Lambda for all three writes: splitting them across services creates a window where the new definition exists but superseded_by is still null — inconsistent staleness detection. Atomicity requires a single transaction.
EB
EventBridge (Custom Bus)

Decouples the Metadata Writer from all downstream consumers. Rules: DefinitionCorrected → DAG Traversal Lambda (always) and DefinitionCorrected → CloudWatch Logs (audit trail). Adding new consumers requires no changes to the publisher.

Why EventBridge over SNS: built-in event replay. If DAG Traversal fails, you can replay from the archive without re-running the correction. SNS has weaker filtering and no replay.
λ
Lambda: DAG Traversal

Responds to DefinitionCorrected. Finds all affected datasets via BFS over dataset_lineage using a PostgreSQL recursive CTE. Computes topological_depth and blocked_by_task_ids for each node. Bulk-inserts all propagation_tasks rows and enqueues only depth-0 tasks to SQS.

For graphs over 10k nodes, BFS is paged in application code with an ElastiCache (Redis) frontier set to avoid reloading visited nodes per iteration.

Why all tasks up front: the full scope is visible before any work starts. Operators can query propagation_tasks immediately. Dynamic task creation makes scope invisible until propagation is halfway done.
Q
SQS: Batch + Streaming Task Queues

Two separate queues because batch and streaming handlers have different timeouts, retry characteristics, and error semantics. Message attributes: tenant_id, task_id, event_id, topological_depth.

  • Batch: 15-minute visibility timeout (Glue job startup + execution)
  • Streaming: 30-minute visibility timeout (drain can be slow)
  • DLQ fires CloudWatch alarm after 3 failed delivery attempts
Why SQS, not Step Functions: the dependency graph is dynamic — varies per correction and per tenant. Step Functions requires the state machine defined at creation time. At 5,000 datasets with multiple state transitions per task, Step Functions cost is materially higher than SQS at $0.40/million messages.
G
Lambda: Batch Handler + Glue / EMR Serverless

Batch Handler sets task status to in_progress, starts a Glue (or EMR Serverless) job parameterized with dataset_id and new_definition_id, then waits for the EventBridge job-state-change event — not a blocking poll. On failure: increments attempt_count; message reappears in SQS up to threshold; then DLQ.

Glue vs EMR Serverless: Glue is the default — no cluster management, pay-per-DPU-second, native catalog integration. EMR Serverless when the team owns PySpark job code. Both publish completion to EventBridge, so handler code is identical.
λ
Lambda: Streaming Task Handler

Most operationally sensitive path. Executes a five-step protocol per streaming dataset:

  • Pause — disable Lambda event source mapping or suspend MSK consumer group
  • Drain — wait for IteratorAgeMilliseconds → 0 (Kinesis) or MSK consumer lag → 0
  • Update — push new definition config to Parameter Store or Glue Schema Registry
  • Replay — reset checkpoint to semantic_field_definitions.created_at of the old version; re-enable event source mapping
  • Resume — consumer reprocesses from replay point under correct semantics

Interim state stored in a separate DynamoDB table — lightweight, ephemeral control-plane state that doesn't need Aurora's relational power.

✓
Lambda: Completion Handler

Called by both handlers on task success. In a single Aurora transaction: marks task completed, updates dataset_field_bindings to the new definition (is_stale=false), increments propagation_events.completed_dataset_count, and closes the event if all tasks are done.

Then queries for tasks whose blocked_by_task_ids are now all completed — enqueues those to the appropriate SQS queue. Wave progression is driven entirely by this DB check; no central scheduler.

Why update bindings here, not in the compute job: the compute job rewrites data but shouldn't own the metadata contract. A single writer keeps the is_stale transition atomic and consistent across all tenants.

Tenant Isolation

Data layer

  • Every row in all five tables carries tenant_id
  • Aurora row-level security policies enforce cross-tenant read prevention (SET app.current_tenant_id at connection time via Lambda middleware)

Compute layer

  • Glue jobs and EMR runs parameterized with tenant_id; write to s3://data-platform/{tenant_id}/datasets/{dataset_id}/
  • IAM execution roles scoped to s3:GetObject / s3:PutObject on the tenant's S3 prefix only

Queue layer

  • Shared queue initially; tenant_id as message attribute with per-tenant Lambda concurrency limits (e.g., 40 executions)
  • Concurrency budget stored in Systems Manager Parameter Store and configurable per tenant
  • Per-tenant dedicated queues available as an upgrade path for hard SLA requirements

Rollback

Rollback is a reverse propagation — no special-casing required

1
Engineer calls POST /definitions/event_timestamp/correct with the old definition's semantics. Metadata Writer inserts a new version row (v3, semantically identical to v1), sets superseded_by on v2, bulk-marks all bindings to v2 as stale, and creates a new propagation_events row with rollback_event_id pointing to the original event.
2
The same EventBridge → DAG Traversal → SQS → Task Handlers → Completion Handler pipeline runs. Datasets recompute against the prior definition.
3
Works because semantic_field_definitions rows are immutable — v1 still exists intact with its full definition. Physical data was never changed during the original correction. Rollback is a metadata pointer swap, not a data recovery operation. Partial rollback (single tenant) is supported by scoping the staleness sweep and task creation to one tenant_id.

Observability

CloudWatch Custom Metrics

  • PropagationTasksCompleted — per event_id, per tenant_id
  • PropagationTasksFailed — per event_id, per tenant_id
  • PropagationLagSeconds — initiated_at to last task completion
  • StreamingDrainDurationSeconds — detect stuck drains

CloudWatch Alarms

  • DLQ message count > 0 → PagerDuty/SNS (stuck or repeatedly-failing task)
  • PropagationLagSeconds > threshold → behind SLA warning
  • Aurora writer CPU > 80% → scale up alert

Status API + Audit Trail

  • GET /propagations/{event_id} — returns overall progress, per-dataset task status, and error messages; queries propagation_events and propagation_tasks
  • All DefinitionCorrected events archived to CloudWatch Logs and optionally S3 via Firehose — full audit log queryable without touching the operational database
