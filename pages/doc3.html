Phase 1 — Submitting the Correction

Engineer calls the API. Three writes, one transaction. One event published.

1

API call

The engineer submits a correction for event_timestamp — previously documented as free_text_string, actually iso8601_date (YYYY-MM-DD, UTC, no time component).

POST /definitions/event_timestamp/correct
{
  "semantic_type":       "iso8601_date",
  "description":         "ISO 8601 calendar date in UTC, no time component (YYYY-MM-DD).",
  "time_semantics":      { "timezone": "UTC", "precision": "day", "format": "YYYY-MM-DD" },
  "nullability_rule":    "event_did_not_occur",
  "uniqueness_guarantee": false
}

API Gateway validates the request body schema before the Lambda is invoked.

2

Insert new definition row

Metadata Writer Lambda opens a single Aurora transaction. First write: insert into semantic_field_definitions.

INSERT INTO semantic_field_definitions (
  definition_id, field_name, version, semantic_type,
  description, time_semantics, nullability_rule,
  uniqueness_guarantee, created_at, created_by, superseded_by
) VALUES (
  '<new-uuid-v2>',   -- generated by Lambda
  'event_timestamp',
  2,                  -- monotonically incremented per (field_name, source_dataset_id)
  'iso8601_date',
  'ISO 8601 calendar date in UTC...',
  '{"timezone":"UTC","precision":"day","format":"YYYY-MM-DD"}',
  'event_did_not_occur',
  false,
  now(),
  'engineer@company.com',
  NULL               -- this is now the current version
)
3

Retire the old definition row

Second write (same transaction): set superseded_by on the version-1 row. The old row is never deleted — it is the permanent record of what was believed to be true before the correction.

UPDATE semantic_field_definitions
SET superseded_by = '<new-uuid-v2>'
WHERE definition_id = '<old-uuid-v1>'
4

Bulk-mark bindings as stale

Third write (same transaction): one statement marks every dataset that was computed against the wrong definition. May touch thousands of rows; runs atomically inside the open transaction.

UPDATE dataset_field_bindings
SET   is_stale = true
WHERE definition_id = '<old-uuid-v1>'

Transaction commits. The DB is now consistent: new definition exists, old one is retired, every dataset that used the wrong definition is marked stale. Nothing downstream has run yet.

5

Publish DefinitionCorrected event

After the transaction commits, the Lambda publishes to the EventBridge custom bus. The event is published outside the transaction — if the publish fails, retrying is safe because the definition write is idempotent against an existing definition_id.

{
  "source":      "walktru.semantic-catalog",
  "detail-type": "DefinitionCorrected",
  "detail": {
    "field_name":        "event_timestamp",
    "old_definition_id": "<uuid-v1>",
    "new_definition_id": "<uuid-v2>",
    "triggered_by":     "engineer@company.com",
    "initiated_at":     "2026-06-23T10:00:00Z"
  }
}
2

Phase 2 — Impact Assessment

EventBridge routes the event to the DAG Traversal Lambda. Which datasets are affected, and in what order?

1

Find directly affected datasets (depth 0)

Query dataset_field_bindings for all datasets that directly reference event_timestamp under the old definition. This is the starting set S₀ — the initial frontier for graph traversal.

SELECT dataset_id
FROM   dataset_field_bindings
WHERE  definition_id = '<old-uuid-v1>'
2

BFS over dataset_lineage to find transitive dependents

Starting from S₀, the Lambda walks downstream through dataset_lineage using a recursive CTE. A billing deduplication dataset that depends on a DAU metrics dataset that depends on event_timestamp appears at depth 2.

WITH RECURSIVE downstream AS (
  SELECT downstream_dataset_id, 1 AS depth
  FROM   dataset_lineage
  WHERE  upstream_dataset_id IN (<S₀ ids>)

  UNION ALL

  SELECT l.downstream_dataset_id, d.depth + 1
  FROM   dataset_lineage l
  JOIN   downstream d ON l.upstream_dataset_id = d.downstream_dataset_id
)
SELECT * FROM downstream

For graphs over 10k nodes, BFS is paged in application code using an ElastiCache (Redis) frontier set — but the result is identical: a flat list of (dataset_id, depth) pairs covering the entire affected subgraph.

3

Compute topological order and blocking relationships

For each affected dataset, compute:

  • ·topological_depth — its depth from the BFS above
  • ·blocked_by_task_ids — task IDs of its direct upstream predecessors within the affected set. Datasets outside the affected subgraph are already correct and need not complete first.

A depth-2 dataset is blocked by its depth-1 predecessors in the affected set — not by all depth-1 datasets globally.

4

Write the propagation plan to the database

The Lambda bulk-inserts the full propagation plan:

1. One row in propagation_events: anchors old_definition_id, new_definition_id, sets affected_dataset_count and status = 'in_progress'.

2. One row per affected dataset in propagation_tasks: topological_depth, blocked_by_task_ids, pipeline_type copied from the datasets table. Status is 'pending' for depth-0 tasks, 'blocked' for all others.

3. Enqueues only the depth-0 tasks to SQS. All deeper tasks stay in the database as blocked — untouched until their upstream tasks complete.

An operator can now query propagation_tasks to see the full scope — e.g., 347 datasets, their depths, and their blocking relationships — before a single recomputation job has run.

3

Phase 3 — Recomputation

Depth-0 tasks run in parallel. Batch and streaming pipelines follow different protocols.

Batch datasets

1

Set in_progress

Batch Handler sets propagation_tasks.status = 'in_progress', records started_at.

2

Start Glue / EMR job

Job parameterized with dataset_id and new_definition_id. Reads the same upstream sources but interprets event_timestamp as an ISO 8601 date — affecting date-range filtering, timezone handling, and aggregation logic. Rewrites materialization to s3://data-platform/{tenant_id}/datasets/{dataset_id}/.

3

Await EventBridge completion event

The handler does not poll. The SQS visibility timeout keeps the message invisible while the job runs. Glue publishes a job-state-change event to EventBridge on success or failure, which triggers the next step.

On failure: attempt_count incremented; message reappears in SQS after timeout. At 3 attempts: task marked failed, message moved to DLQ, CloudWatch alarm fires.

Streaming datasets

1
Pause — disable the Lambda event source mapping (Kinesis) or suspend the MSK consumer group. The consumer stops pulling new records. Events continue arriving in the stream; they accumulate but are not processed.
2
Drain — wait for the in-flight batch to fully commit. For Kinesis: monitor IteratorAgeMilliseconds until it drops to zero. For MSK: consumer lag drops to zero. Interim state written to a DynamoDB control-plane table (consumer_id, pause_timestamp, drain_confirmed_at).
3
Update — push the new semantic definition config to the consumer. In practice: update a Parameter Store value the consumer reads at startup (e.g., event_timestamp.semantic_type = iso8601_date), or update the Glue Schema Registry schema version.
4
Replay — reset the consumer's checkpoint to semantic_field_definitions.created_at for the version-1 row — the moment the wrong definition first became active. Records from that point onward were processed under the wrong definition and must be reprocessed. Re-enable the event source mapping.
5
Resume — consumer processes records from the replay point with the corrected semantic interpretation. It catches up to the current stream position and resumes normal real-time processing.
!

Why replay from definition activation time, not pause time

The pause time only covers records processed since the correction was initiated. But records processed before the correction were also interpreted under the wrong definition — DAU metric aggregations may have bucketed dates incorrectly for months. Replaying from the original created_at ensures the entire historical window under the wrong definition is reprocessed.

4

Phase 4 — Completion and Wave Progression

Each completed task updates the DB and may release the next wave. No central scheduler.

Completion Handler — called on every task success

1
In a single Aurora transaction: set propagation_tasks.status = 'completed', completed_at = now(). Update dataset_field_bindings for this dataset: definition_id = <new_definition_id>, is_stale = false, bound_at = now(). This is the moment the dataset officially transitions from the old semantic contract to the new one. Increment propagation_events.completed_dataset_count.
2
If completed_dataset_count = affected_dataset_count: set propagation_events.status = 'completed', record completed_at. The propagation is done.
3
Query propagation_tasks for any task in blocked status whose blocked_by_task_ids are now all completed. For each: set status to pending, enqueue to the appropriate SQS queue (batch or streaming). This is "wave progression" — no task is ever enqueued until all its upstream dependencies within the affected subgraph have completed.
⚡

Partial Inconsistency During Propagation

While propagation is in progress, some datasets are on the new definition and some are not. This window is unavoidable — and managed explicitly.

Topological guarantee

Recomputation order ensures a downstream dataset is never recomputed before its upstream dependencies. At any point during propagation, a dataset bound to the new definition is guaranteed to have been computed from upstream datasets also on the new definition (or outside the affected subgraph). A depth-2 dataset is never on the new definition while its depth-1 upstream is still on the old one.

Warning mode (default)

Queries against stale datasets succeed but return metadata indicating the dataset was computed against an older semantic definition. Analysts see results but are warned they may not reflect corrected event_timestamp semantics. Appropriate for most analytical workloads.

Blocking mode (opt-in)

Queries against stale datasets are rejected until recomputation completes. Appropriate for billing or compliance datasets where reading a stale version is worse than reading nothing. Configurable per tenant via the is_stale flag visible to the query layer.

No phantom consistency

propagation_events.status stays in_progress until every task completes. The Status API (GET /propagations/{event_id}) reflects real-time progress as a rolling count of completed_dataset_count / affected_dataset_count. The system does not pretend propagation is complete until it is.

Rollback

Rollback is not a special operation — it is a new correction in the reverse direction.

Detecting the need to roll back

!
CloudWatch alarm on DLQ message count > 0 (task failing repeatedly)
!
CloudWatch alarm on PropagationLagSeconds exceeding SLA threshold
!
Manual call to GET /propagations/{event_id} showing stuck tasks or tenants reporting unexpected results

Executing rollback

1
Engineer calls POST /definitions/event_timestamp/correct again with the old definition's semantics. Metadata Writer inserts v3 (semantically identical to v1), sets superseded_by on v2, bulk-marks bindings to v2 as stale, and creates a new propagation_events row with rollback_event_id pointing to the original event.
2
The same EventBridge → DAG Traversal → SQS → Task Handlers → Completion Handler pipeline runs. Datasets recompute against the prior definition (v3 / restored v1 semantics).
3
Physical data was never modified — Glue and streaming jobs rewrote materialized outputs to S3, but the original pre-correction outputs are not deleted until new outputs are verified. Because semantic_field_definitions rows are immutable, v1 still exists in full detail. Rollback is a metadata pointer swap, not data recovery.
4
Partial rollback (single tenant): scope the UPDATE dataset_field_bindings SET is_stale=true to a specific tenant_id in the Metadata Writer. The DAG Traversal Lambda creates tasks only for that tenant's affected datasets. Other tenants' propagation continues uninterrupted.
⚖

Tradeoffs

Three design choices where the alternative was viable — here's why each call was made.

Correctness vs cost

Eager full recomputation vs lazy on-read re-evaluation

Chosen — eagerEvery affected dataset is recomputed immediately. The propagation has a clear completion state. Operators know when the system is fully consistent. Rollback scope is well-defined.
vs
Lazy (rejected)Stale datasets are only recomputed when a tenant queries them. Saves compute cost. But propagation has no completion state, rollback is harder (you don't know what's been updated), and billing datasets silently serve stale data.

For a platform where event_timestamp feeds billing deduplication and identity joins, the risk of silent staleness outweighs the compute cost.

Versioned definitions vs in-place updates

Append-only rows vs overwriting the definition row

Chosen — immutable versioningEach correction adds a new row. Old rows are never modified. You can always ask "what definition was this dataset computed against?" Rollback is a metadata pointer swap — no data recovery needed.
vs
In-place update (rejected)Simpler to implement. But eliminates the ability to audit past definitions. Rollback becomes destructive — you've overwritten the thing you'd need to restore.

Cost: one row per definition version (trivial storage), slightly more complex "get current definition" query. Both trivially acceptable.

Up-front task set vs dynamic task spawning

All tasks inserted at creation time vs children spawned at completion

Chosen — up frontFull scope is visible before any task runs. Operators can estimate total time and audit impact. Easy to catch DAG cycles or unexpectedly large impact sets before recomputation starts. Complete task list simplifies rollback.
vs
Dynamic spawning (rejected)Simpler to implement — a task just creates its children on completion. But scope is opaque until propagation is halfway done, with no way to estimate how long it will take.
