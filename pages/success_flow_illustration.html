Successful Semantic Update - Table Flow

Only the six metadata tables and their state transitions

This illustrates what happens inside the six metadata tables when a new semantic definition is published successfully, including the dataset read-pointer promotion. It intentionally excludes AWS services, queues, jobs, and operational failure branches.

Definition State

  • semantic_field_definitions has both v1 and v2.
  • v1 keeps its old semantic contents but points to v2 via superseded_by.
  • v2 is the current semantic definition for event_timestamp.

Planning State

  • dataset_lineage is read, not changed.
  • propagation_events stores the overall correction event.
  • propagation_tasks stores every affected dataset and its blockers.

Completion State

  • Completed tasks promote the dataset read pointer in datasets.
  • Direct bindings move from v1 to v2 and clear is_stale.
  • Transitive-only tasks complete without creating fake semantic bindings.

Successful Sequence

  1. Insert the new semantic version in semantic_field_definitions and mark the prior version as superseded.
  2. Mark direct dataset_field_bindings rows that used the prior definition as stale.
  3. Read direct stale bindings, then read dataset_lineage to find all transitive downstream datasets.
  4. Create one propagation_events row and one propagation_tasks row for each affected dataset.
  5. As each dataset successfully recomputes, mark its task completed, promote its materialization pointer, and update direct bindings to the new definition.
  6. When all tasks are completed, mark the propagation event completed.
